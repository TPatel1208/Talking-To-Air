# PRD T32 — Metadata tab redesign: Overview/Details views for chart artifacts

**Repo:** Talking-To-Air · **Session scope:** one session, one commit · **Label:** ready-for-agent
**Depends on:** none hard (builds on existing `ProvenanceBlock`/`OutputPanel`, `AggregationService`, `mask_info`/`qa_flags`, all already merged). Soft relation to T10 (provenance pane vision) — see Further Notes. Decision record 2026-07-14 (grilling session).

## Problem Statement

The Metadata tab is the only place a user can check what they're actually looking at, and today it's thin, buggy, and chart-only. `ProvenanceBlock` (`Frontend/src/components/ChartMessage.jsx:184-219`) renders five fields in one flat, ungrouped strip — but two of them are silently broken: `provenance.dataset` is never set by the backend, so "Dataset" always falls back to showing the variable name; `provenance.source`/`endpoint` are never set for satellite charts either, so "Source" never renders at all. Timeseries charts don't get `aggregation_meta` attached (`Backend/tools/satellite_tools/plot_tools.py:945-953`), so their Granules block never appears even though masking info exists for them. Meanwhile real, already-computed context — dataset short_name/version/description, provider/instrument, QA methodology (not just what was applied), variable long_name/advisory notes/valid ranges, and the query/handles needed to reproduce a result — never reaches the UI at all. A researcher trying to judge "can I trust this" or "how do I cite/reproduce this" gets none of it from this tab.

## Solution

Split the Metadata tab into two views: **Overview** (short, fully expanded, answers "what am I looking at / where's it from / can I trust it") and **Details** (grouped, collapsible sections plus a raw-JSON escape hatch, for anyone who wants the full picture). Fix the backend provenance builder to attach a real `dataset` field (registry short_name/description, distinct from variable), a real `source` field (provider + instrument, e.g. "NASA GES DISC — MODIS/Terra"), and wire `aggregation_meta` through for timeseries charts so they get the same granule/cadence treatment as other chart types. Attach variable-definition facts (long_name, advisory notes, valid ranges, mask methodology — currently locked inside the separate `describe_dataset` MCP tool's chat-text output) directly onto the chart payload at creation time, since the backend already resolves them from the registry/UMM-Var lookup during chart generation. Add a condensed, color-coded QA/masking status line to Overview that points to the Statistics tab's existing `MaskingDisclosure` for the full breakdown, rather than duplicating its logic. Add low-emphasis "view source dataset" link and "copy citation" affordances, and a Details-only "Reproducibility" section (query parameters + source handles, "copy query" action) that is deliberately separate from citation — one is about citing the dataset in general, the other about reproducing this specific request.

## User Stories

1. As a general user, I want a short Overview that tells me what dataset/variable I'm looking at, the exact time range and region of *this* request, and a quick trust signal for data quality, so that I don't have to parse raw metadata to answer "what am I looking at."
2. As a general user, I want the Overview to clearly separate facts about *this specific request* from facts about the *source dataset as a whole*, so that I don't mistake the collection's full coverage for what's actually rendered on screen.
3. As a general user, I want missing fields to say "Not available" rather than disappear silently, so that I know a fact was checked for and doesn't exist, not simply forgotten by the UI.
4. As a general user, I want a colored status indicator alongside the QA/masking summary, so that I can gauge data quality at a glance before reading details.
5. As a researcher, I want a Details view with grouped sections (Spatial, Temporal, Provenance, Variable Definition, Reproducibility) so that I can find one specific fact quickly instead of scanning a flat list.
6. As a researcher, I want variable-level facts — long_name, advisory notes, valid ranges, mask methodology — surfaced directly in Details, so that I don't need to run a separate `describe_dataset` query to understand what I'm plotting.
7. As a researcher, I want a "copy citation" action and a link to the dataset's authoritative page, so that I can cite what I used without leaving the app.
8. As a researcher, I want a "Reproducibility" section with the exact query parameters and source handles behind this result, plus a "copy query" action, so that I can rerun or share this exact request.
9. As a researcher, I want timeseries charts to show granule count and cadence like other chart types, so that "how much data went into this line" isn't only answered for some chart types.
10. As a researcher, I want the full list of individual granule dates available (in Details, not cluttering Overview), so that I can audit exactly which files contributed without losing the tab's scannability.
11. As a developer, I want the backend to attach `dataset` (registry short_name/description) and a real `source` (provider + instrument) onto provenance, so that the frontend never has to guess or silently mislabel a fallback value again.
12. As a developer, I want variable-definition facts attached at chart-creation time rather than fetched lazily, so that expanding the Details section never needs a loading state for data the backend already had in memory.

## Implementation Decisions

- **Two-view structure.** `MetadataTab` renders an `Overview`/`Details` sub-tab pair (simple local toggle state, no routing). Overview is always fully expanded; Details uses per-section `<details>`/accordion components, collapsed by default, following the existing `MaskingDisclosure`/`StatCard` visual convention (`OutputPanel.jsx:58-112`) for consistency with the Statistics tab.
- **Overview content, in order:** (1) "This view" — dataset, variable, date range, region, aggregation, and a one-line granule summary (count + cadence, no date list); (2) QA/trust status line — colored dot (green/yellow/red for clean/masked/ambiguous) + short text (e.g. "12% masked — cloud QA flag"), sourced via the existing `resolveMasking` util, with a "see Statistics tab for details" pointer — no re-implementation of `MaskingDisclosure`'s logic; (3) "Source dataset" — collection short_name/description/version, provider + instrument, with a low-emphasis outbound link to the authoritative dataset page and a "copy citation" button.
- **Details content:** grouped sections — Spatial (bbox, region name), Temporal (date range, cadence, full expandable granule date list), Provenance (QA methodology fields — `quality_flag_var`, `qa_good_values`/`qa_bad_values`, fill/valid-range sources — as general methodology, distinct from Overview's "what was actually applied" line), Variable Definition (long_name, units, advisory notes, valid ranges, mask note), Reproducibility (query snapshot: dataset/start/end/bbox/aggregation/chart_parameters, source_handles, "copy query" button). A "view raw JSON" toggle at the bottom of Details renders the full provenance/aggregation_meta object for power users.
- **Backend: `dataset` field.** `_provenance()` (`Backend/tools/satellite_tools/plot_tools.py:458-480`) gains a `dataset` key sourced from the registry's `short_name`/`description` (`Backend/datasets/registry.py`) for the resolved collection — distinct from `variable`, which keeps its current meaning.
- **Backend: `source` field.** `_provenance()` sets `source` to `"<provider> — <instrument/platform>"` where available from the registry/collection config (e.g. "NASA GES DISC — MODIS/Terra"); access-method detail (Harmony vs. OPeNDAP) is not part of this field — it belongs in Reproducibility if ever needed, not Overview.
- **Backend: timeseries `aggregation_meta`.** The timeseries chart path (`plot_tools.py:945-953`) is updated to call `_attach_reproducibility` with the same `agg_meta` construction other chart types already get, so Granules/cadence render consistently. `ts_payload["masking"]` (already present at line 944) is threaded the same way non-timeseries masking is.
- **Backend: variable-definition attach.** At chart-creation time, the backend copies the UMM-Var/registry facts it already resolves (long_name, advisory_notes, fill_values, valid_ranges, mask_note — currently only exposed via the separate `describe_dataset` MCP tool per `Backend/earthdata_mcp/workspace.py:204-238`) onto the provenance/payload dict, so Details' Variable Definition section renders with zero additional round-trips.
- **Missing fields.** Every field defined for a section renders even when empty, labeled "Not available" — this is a deliberate reversal of `ProvenanceBlock`'s current `.filter(([, value]) => value)` behavior (`ChartMessage.jsx` filters empty entries out entirely today).
- **Scope: chart artifacts only.** `MetadataTab`'s `artifact` branch stays dead in this PRD — map/table/comparison artifacts do not get a Metadata tab yet (planned T33). Compare-mode side-by-side rendering of two datasets' metadata is also out of scope here (planned T34, depends on this PRD's Overview/Details component existing first).

## Testing Decisions

- Backend (`docker compose --profile test run --build --rm backend-test`): `_provenance()` unit tests asserting `dataset` and `source` are populated from registry fixtures distinct from `variable`; a timeseries-path test asserting `aggregation_meta`/granule fields are now attached identically to the non-timeseries path; a variable-definition-attach test asserting long_name/advisory_notes/valid_ranges appear on the payload without a second MCP call (mock/spy the registry lookup, assert call count unchanged from today).
- Frontend (`docker compose --profile test run --build --rm frontend-test`): Overview renders "This view"/QA status/"Source dataset" from a fixture provenance object; Details renders all five grouped sections collapsed by default and expands independently; a field fixture with an empty value renders "Not available" rather than being omitted; granule date list appears only in Details, granule count+cadence appears only in Overview; "copy citation" and "copy query" buttons write the expected string to clipboard (mocked); raw-JSON toggle renders the underlying object verbatim.
- Live verification: run a satellite chart query in the browser, open Metadata → Overview, confirm Dataset/Source now render real values (not blank/mislabeled); switch to Details, expand each section, confirm Variable Definition and Reproducibility populate; run a timeseries query and confirm its Granules line now appears.

## Out of Scope

- Extending the Metadata tab to non-chart artifact outputs — planned as **T33**. Note: T33's investigation found this narrower than it first appeared — only `table` and previously-unreachable ground-validation `timeseries` artifacts actually need new Overview/Details; `map`/`comparison`/chart-backed-`timeseries` artifact stubs are deliberately left un-carded since their full data already reaches users via this PRD's chart tabs.
- Compare-mode side-by-side metadata rendering — planned as **T34**. Depends only on this PRD (T32), not on T33 — compare mode is chart-kind only and never touches artifact-type data. No auto-diffing between compared datasets' metadata (agreed: plain side-by-side only, when built).
- T10's full provenance-pane vision: MCP-side lineage/ancestry graph, deterministic Markdown/Word methods-text export, CSV/NetCDF data download. This PRD's "copy citation" is a lightweight, UI-only affordance (link + string), not T10's MCP citation-tool integration — they can coexist; T10 remains the larger future vision for formal, paper-grade provenance.
- Any change to what masking/QA computation does — only what's *displayed* changes here; `resolve_mask_info`/`resolve_qa_info` (`Backend/datasets/mask_info.py`, `Backend/datasets/qa_flags.py`) are read, not modified.

## Further Notes

This PRD intentionally stops short of T10: T10 envisions a dedicated fourth workbench pane with MCP-side lineage walks and a deterministic methods-generator for publication use; this PRD is a near-term fix to an existing, broken, chart-only tab. Where they'd overlap (citations), this PRD's version is a simple link + copyable citation string sourced from whatever the registry already has — not T10's MCP citation-tool round trip. When T10 is eventually built, its citation feature can supersede this one without any structural conflict, since this PRD doesn't invent a competing citation data model.

The dead `artifact` branch in `MetadataTab` (`OutputPanel.jsx:161-169`, never reached because the call site at line 406 only passes `chart`) is left in place rather than deleted — T33 wires it up for table and ground-validation-timeseries artifacts specifically (not all artifact types uniformly; see T33's Problem Statement), so removing it now would just mean re-adding the same shape shortly after.
